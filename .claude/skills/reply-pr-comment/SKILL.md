# Reply to Pull Request Comment

PRについたレビューコメントを確認し、適切に対応・返信する。

## 手順

### 1. PRコメントの確認

コメントの種類に応じて以下のコマンドを使い分ける:

- **PR全体へのコメント**: `gh pr view --json comments --jq '.comments[] | {author: .author.login, body}'`
- **コード行に紐づいたレビューコメント**: `gh api repos/{owner}/{repo}/pulls/{number}/comments --jq '.[] | {path, line, body, user: .user.login}'`
- **ブラウザで確認**: `gh pr view --web`

未対応のコメントを特定し、コメントの内容と意図を正確に理解する

### 2. コメント内容の検討

- 指摘が正しいかどうかを確認する
- 指摘内容に疑問がある場合は、対応前にユーザーに確認する
- 必要に応じてコードの該当箇所を確認する

### 3. コード修正（必要な場合）

- 指摘に基づいてコードを修正する
- 修正は最小限に留め、指摘の範囲内で対応する
- 修正後、関連するテストを実行して問題がないことを確認する

### 4. コミットとプッシュ

- **コミット・プッシュ前に必ずユーザーに確認を取る**
- 修正内容をユーザーに提示し、コミットしてよいか確認する
- 確認が取れたら修正内容をコミットする
  - コミットメッセージ: `fix: <指摘内容の要約>` または `refactor: <改善内容>`
  - レビューコメント単位でコミットを分ける
- プッシュしてよいか確認を取り、`git push` でリモートにプッシュする

### 5. コメントへの返信

- **返信前に必ずユーザーに返信内容を提示し、確認を取る**
- 確認が取れたらコメントに返信する
- コメントの種類に応じて適切な返信方法を選択する
  - **PR全体へのコメント**: `gh pr comment` コマンドを使用してPR全体に返信する
  - **コード行に紐づいたレビューコメント**: `gh api repos/{owner}/{repo}/pulls/{number}/comments --method POST --field body='返信内容' --field in_reply_to={review_id}` コマンドを使用して該当行に返信する
- 返信内容には以下を含める:
  - 対応した場合: 修正内容の簡潔な説明と該当する修正の入ったコミットへの参照（コミットハッシュ）
  - 質問がある場合: 具体的な質問内容
  - 対応しない場合: 理由の説明
- 返信は日本語で簡潔に記述する

## 返信テンプレート

### 修正対応した場合

```
ご指摘ありがとうございます。修正しました。
<修正内容の簡潔な説明>

<コミットハッシュ>
```

### 質問・確認がある場合

```
確認させてください。
<具体的な質問内容>
```

### 対応しない場合（理由あり）

```
ご指摘ありがとうございます。
以下の理由から、今回は現状維持としたいと考えています。
- <理由>

ご意見いただけますと幸いです。
```

## 注意事項

- レビューコメントの指摘が正しいことを確認してから対応すること。
- 指摘内容に疑問がある場合は必ず質問すること。
- 修正後は必ずテストを実行して確認すること。
- 1つのコメントに対して1つのコミットを心がける（関連する複数の指摘は1コミットにまとめても可）。
- 返信は具体的かつ簡潔に書くこと。
